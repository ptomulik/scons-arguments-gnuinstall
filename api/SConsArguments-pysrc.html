Source Code for Package SConsArguments

  1  """`SConsArguments` 
  2   
  3  **Intro** 
  4   
  5  This package implements SCons *arguments*. A SCons *argument* is an entity 
  6  which correlates up to three *endpoints*: 
  7   
  8  - single construction variable in SCons environment (``env['NAME'], env.subst('$NAME')``), 
  9  - single SCons command-line variable (``scons variable=value`` in command-line), and 
 10  - single SCons command-line option (``scons --option=value`` in command-line). 
 11   
 12  Some of the above may be missing in *argument*'s specification, so we may for 
 13  example correlate only a construction variable with a command-line option 
 14  without involving command-line variable. *Arguments* specify how information 
 15  shall flow from command-line to SCons environment. 
 16   
 17  **Endpoint names and data flow** 
 18   
 19  Each *argument* has up to three *endpoints*: 
 20   
 21  - ``ENV`` *endpoint*: a construction variable in SCons environment, 
 22  - ``VAR`` *endpoint*: a command line variable, and 
 23  - ``OPT`` *endpoint*: a command line option. 
 24   
 25  Separate "namespaces" are used to keep names of ``ENV``, ``VAR`` and ``OPT`` 
 26  endpoints (i.e. construction variables, command-line variables and command-line 
 27  options). The user defines mappings between *endpoints* when specifying 
 28  *arguments*. *Arguments* also have their own names which may be independent of 
 29  their endpoint names. For example, one may create an *argument* named ``foo`` 
 30  which correlates a construction variable named ``ENV_FOO``, command-line 
 31  variable named ``VAR_FOO`` and command-line option identified by key 
 32  ``opt_foo`` (we use ``dest`` attribute of command line option as its 
 33  identifying key, see `option attributes`_ of python ``optparse``). At certain 
 34  point *arguments* get requested to update SCons environment ``env``, that is 
 35  to populate environment with values taken from command-line variables and/or 
 36  options.  At this point, value taken from command-line variable ``VAR_FOO`` or 
 37  value from command-line option ``opt_foo`` is passed to construction variable 
 38  ``ENV_FOO``. If both,command-line variable and command-line option are set, 
 39  then command-line option takes precedence. 
 40   
 41  **Substitutions in Arguments** 
 42   
 43  If a command-line value is a string, it may contain placeholders (e.g. 
 44  ``VAR_FOO`` may be a string in form ``"bleah bleah ${VAR_BAR}"``, which contains 
 45  placeholder ``${VAR_BAR}``). The placeholder is assumed to be the name of 
 46  *endpoint* from the same namespace where the placeholder appears. It means, 
 47  that if we have a command-line variable, and its value is a string containing 
 48  placeholder ``"$VVV"``, then ``VVV`` is assumed to be the name of another 
 49  command-line variable (and not, for example, construction variable). When 
 50  passing strings from command-line variables and options to a SCons environment, 
 51  the placeholders are renamed such that they refer to corresponding construction 
 52  variables in SCons environment. This is shown in the example below. 
 53   
 54  **Example** 
 55   
 56  Assume, we have the following three *arguments* defined:: 
 57   
 58      .               (1)         (2)         (3) 
 59      Arguments:      foo         bar         geez 
 60      Environment:    ENV_FOO     ENV_BAR     ENV_GEEZ 
 61      Variables:      VAR_FOO     VAR_BAR     VAR_GEEZ 
 62      Options:        opt_foo     opt_bar     opt_geez 
 63      .             --opt-foo   --opt-bar   --opt-geez 
 64   
 65   
 66  and we invoked scons as follows:: 
 67   
 68      # Command line: 
 69      scons VAR_FOO='${VAR_BAR}' VAR_BAR='${foo}' --opt-geez='${opt_foo}' 
 70   
 71  then, after updating a SCons environment ``env`` with *arguments*, the 
 72  environment shall have the following construction variables set:: 
 73   
 74      env['ENV_FOO'] = '${ENV_BAR}'   # VAR_FOO -> ENV_FOO,  VAR_BAR -> ENV_BAR 
 75      env['ENV_BAR'] = '${foo}'       # VAR_BAR -> ENV_BAR,  foo -x-> foo 
 76      env['ENV_GEEZ'] = '${ENV_FOO}'  # opt_geez-> ENV_GEEZ, opt_foo -> ENV_FOO 
 77   
 78  The arrow ``-x->`` denotes the fact, that there was no command-line variable 
 79  named ``foo``, so the ``"${foo}"`` placeholder was left unaltered. 
 80   
 81  **Example** 
 82   
 83  The following ``SConstruct`` file defines three *arguments*: ``foo``, ``bar`` 
 84  and ``geez``. Corresponding construction variables (environment) are named 
 85  ``ENV_FOO``, ``ENV_BAR`` and ``ENV_GEEZ`` respectively. Corresponding 
 86  command-line variables are: ``VAR_FOO``, ``VAR_BAR`` and ``VAR_GEEZ``. Finally, 
 87  the command-line options that correspond to our *arguments* are named 
 88  ``opt_foo``, ``opt_bar`` and ``opt_geez`` (note: these are actually keys 
 89  identifying options within SCons script, they may be different from the option 
 90  names that user sees on his screen - here we have key ``opt_foo`` and 
 91  command-line option ``--foo``). 
 92   
 93  .. python:: 
 94   
 95      from SConsArguments import ArgumentDeclarations 
 96      env = Environment() 
 97      decls = ArgumentDeclarations( 
 98         # Argument 'foo' 
 99         foo = (   {'ENV_FOO' : 'ENV_FOO default'},                   # ENV 
100                   ('VAR_FOO',  'VAR_FOO help'),                      # VAR 
101                   ('--foo', {'dest' : "opt_foo"})         ),         # OPT 
102         # Argument 'bar' 
103         bar = (   {'ENV_BAR' : None},                                # ENV 
104                   ('VAR_BAR', 'VAR_BAR help', 'VAR_BAR default'),    # VAR 
105                   ('--bar',  {'dest':"opt_bar", "type":"string"})),  # OPT 
106         # Argument 'geez' 
107         geez =(   {'ENV_GEEZ' : None},                               # ENV 
108                   ('VAR_GEEZ', 'VAR_GEEZ help', 'VAR_GEEZ default'), # VAR 
109                   ('--geez', {'dest':"opt_geez", "type":"string"}))  # OPT 
110      ) 
111      variables = Variables() 
112      args = decls.Commit(env, variables, True) 
113      args.UpdateEnvironment(env, variables, True) 
114   
115      print("env['ENV_FOO']: %r" %  env['ENV_FOO']) 
116      print("env['ENV_BAR']: %r" %  env['ENV_BAR']) 
117      print("env['ENV_GEEZ']: %r" %  env['ENV_GEEZ']) 
118   
119  Running scons several times for this example, different results may be obtained 
120  depending on command-line variables and options provided. Let's do some 
121  experiments, first show the help message to see available command-line options:: 
122   
123      user@host:$ scons -Q -h 
124      env['ENV_FOO']: 'ENV_FOO default' 
125      env['ENV_BAR']: 'VAR_BAR default' 
126      env['ENV_GEEZ']: 'VAR_GEEZ default' 
127      usage: scons [OPTION] [TARGET] ... 
128   
129      SCons Options: 
130         <.... lot of output here ...> 
131      Local Options: 
132        --geez=OPT_GEEZ 
133        --foo=OPT_FOO 
134        --bar=OPT_BAR 
135   
136  then play with them a little bit (as well as with command-line variables):: 
137   
138      user@host:$ scons -Q --foo='OPT FOO' 
139      env['ENV_FOO']: 'OPT FOO' 
140      env['ENV_BAR']: 'VAR_BAR default' 
141      env['ENV_GEEZ']: 'VAR_GEEZ default' 
142      scons: `.' is up to date. 
143   
144      user@host:$ scons -Q VAR_FOO='VAR_FOO cmdline' 
145      env['ENV_FOO']: 'VAR_FOO cmdline' 
146      env['ENV_BAR']: 'VAR_BAR default' 
147      env['ENV_GEEZ']: 'VAR_GEEZ default' 
148      scons: `.' is up to date. 
149   
150      user@host:$ scons -Q VAR_FOO='VAR_FOO cmdline' --foo='opt_foo cmdline' 
151      env['ENV_FOO']: 'opt_foo cmdline' 
152      env['ENV_BAR']: 'VAR_BAR default' 
153      env['ENV_GEEZ']: 'VAR_GEEZ default' 
154      scons: `.' is up to date. 
155   
156      user@host:$ scons -Q VAR_FOO='VAR_FOO and ${VAR_BAR}' 
157      env['ENV_FOO']: 'VAR_FOO and ${ENV_BAR}' 
158      env['ENV_BAR']: 'VAR_BAR default' 
159      env['ENV_GEEZ']: 'VAR_GEEZ default' 
160      scons: `.' is up to date. 
161   
162      user@host:$ scons -Q --foo='opt_foo with ${opt_geez}' 
163      env['ENV_FOO']: 'opt_foo with ${ENV_GEEZ}' 
164      env['ENV_BAR']: 'VAR_BAR default' 
165      env['ENV_GEEZ']: 'VAR_GEEZ default' 
166      scons: `.' is up to date. 
167   
168  *Arguments* are very flexible and provide much more than presented above. The 
169  documentation of `ArgumentDeclarations()`, `ArgumentDeclaration()`, `DeclareArguments()`, 
170  `DeclareArgument()`, `_ArgumentDeclarations`, `_Arguments`, and `_ArgumentDeclaration` shall 
171  be a good starting point for developers and advanced users. 
172   
173  .. _option attributes: http://docs.python.org/2/library/optparse.html#option-attributes
